## 二、 深度技术分析

### 1. 语义召回的“BGE 效应”
通过切换至 `BGE-Small-zh v1.5`，系统在处理“住院天数”等业务强相关词汇时，检索 P90 耗时从原先的 15s（含加载）压降至 **0.4s**，且召回准确度提升显著。Case 02 的类型转换修复正是由于语义层精准提示了 `ipt_days` 的业务含义。

### 2. 反馈回路的“自愈”能力
在 QA-01 测试中，系统首轮生成的 SQL 被 SQLGuardian 拦截（疑似笛卡尔积）。Agent 在 **0.8s 内** 捕获了错误日志，并自动通过 Prompt 注入了修正建议。在第二轮重试中，模型成功写出了带 `INNER JOIN ON` 的标准 SQL，体现了极强的 **架构自愈力**。

### 3. PII 隐私盾的实测表现
在 QA-03 和 QA-08（涉及患者明细）中，报告中的个人编号被强制执行了正则掩码处理：
*   **原始数据**: `P2021000456`
*   **报告输出**: `P202****456`
符合国家医保局关于“敏感数据不出本地、非必要不脱敏”的合规要求。

## 三、 遗留风险与优化路径

### 🔴 风险：API 配额瓶颈 (429 Error)
**现象**：在连续高强度执行到第 9 个用例时，Doubao API 触发了流量限制（SetLimitExceeded）。
**对策**：
1.  **Failover 机制**：在 `llm_providers.json` 中配置备选 Endpoint。
2.  **本地化部署**：针对 18GB 生产数据，建议将 BGE 和 Coder 模型进行私有化部署。

### 🟡 优化：审计节点焦虑
**现象**：Auditor 节点在面对开发库“空数据”时，容易产生冗余的思考步骤。
**对策**：在 `AUDITOR_PROMPT` 中注入开发库数据样本分布，减少其对“零结果”的过度推演。
